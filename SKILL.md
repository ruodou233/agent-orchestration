# Agent Orchestration — 共享长任务治理规范

## 目标

本文是 Codex、Claude Code 共用的长任务治理规范，用来处理无人值守、批量扫描、批量生成、长文本分析、多子代理并行、上下文窗口接近上限等任务。

它只定义底层工作方法，不绑定某个平台的具体工具。各平台可以保留自己的薄入口和执行适配器。

## 核心原则

### 主代理负责指挥，不负责吞全文

主代理的职责是：

- 和用户确认目标、边界、交付物；
- 清点输入，估算规模和风险；
- 拆分任务并分派给子代理、工具或脚本；
- 收集压缩结果；
- 去重、交叉验证、做最终决策；
- 判断哪些动作已由用户明确授权，哪些因目标或范围不清需要澄清。

在上下文密集型任务里，主代理通常不直接读取全部原文。大文件、长对话、批量文档和跨文档比对优先进入分片或子代理流程。

主代理优先做能优化战略的判断与沟通；看过之后不能改进战略、只会污染上下文的具体执行工作，优先下放给指挥官/子代理。

### 状态落盘，结果压缩

长任务必须把状态写到文件，而不是只存在当前对话里。

推荐最小状态文件：

```text
workspace/task_goal.md
workspace/progress.md
workspace/queue.jsonl
workspace/manifest.jsonl
workspace/sub_agent_outputs/
workspace/failures.md
workspace/handoffs/
```

子代理、shell、工具的完整输出应保存到文件。回传给主代理时只提供摘要、状态、风险和 `output_path`。

### 稳定 ID 和 append-only 事件流

批量任务必须给每个对象和任务建立稳定 ID。

- `manifest.jsonl` 记录对象状态，例如对话、文档、URL、文件、产物；
- `queue.jsonl` 记录任务状态，例如已排队、运行中、完成、失败、超时、跳过；
- 二者都采用 append-only 方式，方便断点续跑、审计和回滚判断。

示例：

```json
{
  "item_id": "conversation_2026_06_07_001",
  "kind": "conversation",
  "status": "scanned",
  "metadata": {
    "source_path": "...",
    "summary_path": "workspace/sub_agent_outputs/conversation_2026_06_07_001.md",
    "size_bytes": 912345,
    "risk": "large_context"
  },
  "timestamp": "2026-06-08T12:00:00"
}
```

### 子代理边界要清楚

每个子代理任务都应有：

- `task_id`；
- 输入范围；
- 预期输出格式；
- 是否允许写文件；
- 超时或停止条件；
- 失败分类；
- 输出保存位置。

子代理之间默认上下文隔离。主代理只整合结果，不让不同子代理互相污染判断。

### 角色与模型按档位分配，不钉版本号

模型按角色/能力档位指定，而不是写死具体型号，模型换代时本规范不必改：

- **主代理**：模型由用户指定。优先与用户讨论战略、确认目标与边界、做最终决策；在长任务中尽量避免亲自执行会污染上下文的具体工作。
- **指挥官（中层管理）**：用当前最强模型。自主决定上下文分配、工人数量、尝试路线、交接时机，以及每个工人的模型与推理档位（reasoning effort）（工人默认落 Codex 侧、主要调 effort；两层架构下此职责归主代理，见工人段）。
- **工人**：执行边界清楚、可验证的具体子任务；默认一律派 Codex/GPT 侧执行（token 充裕、默认强档，难度路由主要调 effort）。Claude 侧不设执行类工人默认档（2026-07-06 起），Claude 侧子代理仅限收敛位/分析/例外表场景（见「Claude Code」段）。各侧命令见「平台适配」；档位策略见 `ai-knowledge-base.md` §3.3（Advisor），effort 手段见 §3.4（该参考文档随 [upgrade-audit](https://github.com/ruodou233/upgrade-audit) 包的 references/ 发布）。结果压缩后回报上一层。
- **收敛位（聚合/验收/评审/仲裁）**：合成多路结果、交付前验收、对比评审的子代理不适用工人降档逻辑，默认取该侧强档——聚合位对档位的敏感度高于 proposer 位（同组输入换聚合者可差 15+ 个百分点，MoA 实测；judge 档位差 10–25 分，Fusion 官方）。自动化脚本中的合成调用必须显式传 model（配置项、不写死型号），不随 CLI 默认漂移。收敛位职能默认留 Claude 侧（验收属主代理职责）。
- **proposer 池构成**：当前模型谱系存在明确强档时，不做多家模型出稿合并（Mixed-MoA，混合会稀释强档输出），多样性用同模型多采样/多 persona 获得；仅当多个同档模型水平接近时，多源 proposer 才可能有净增益。

拆分原则：按独立性拆，不为便宜拆细——收益=并行+上下文隔离+（Claude 侧）保温节拍，成本=每个子代理固定开销（数千 token）+背景重建；五个维度：并行性、上下文隔离、可验证性（低档/低成本执行模型只做可被 schema/测试/diff/上级复核验证的任务）、上下文搬运成本、合并冲突（同一文件/同一决策面不并行拆给多个工人）。调研/信息收集型任务优先按**证据获取轨迹**拆分（不同信息源、搜索策略、工具路径）；需要新证据的调研工人必须配信息获取工具（整理本地已有材料的工人不受此约束）——多路并行的增益主要来自互补证据。可验证任务按验证器收敛，不上共识聚合。纯观点/分析类多样性不单独开多模型工人，用单强模型多 persona（wisdom-roundtable 模式）或多采样即可。fan-out 常规 2–4 个，超过须说明独立性与合并策略。拆细使子任务变简单时档位可下调，下限以该子任务失败重做成本为界。

限制原则：默认不给模型加限制，只有确认某条限制确实带来更好结果时才加。顶级模型尤其不要写成低智能执行器。

具体平台与模型映射属于使用者偏好；作者环境记录在其长期记忆文档（私有，不随包发布），你的映射写入本 skill 的 local-config。

### 编排层级：两层 / 三层

编排可取两层或三层，指挥官层可选；过于简单、不值得编排的请求不在本规范范围。按下列信号决定层数：

- **两层（主代理 → 工人）**：主代理直接派工人并行、自己整合，并承担原属调度方的职责（含交付前独立验收，见该节）。
- **三层（主代理 → 指挥官 → 工人）**：出现以下任一信号即插入指挥官层（职责见角色段）——
  - 并行工人数多到主代理无法在不读全文前提下逐个校验；
  - 需跨多代/多批次迭代，一轮 fan-out 收不完；
  - 交付前验收本身需专职调度，主代理顺手做不了。

两层/三层的层级选择由主代理把关；任何平台自带的自动调度/并行能力都只是执行层，不替代此架构决策。

### 交付前独立验收

负责该块的调度方（三层=指挥官，两层=主代理）在判断某块任务完成、交付或向上一层提交之前，应先启动一个全新的、上下文独立的子代理做验收检查，而不是用执行过该任务、已被上下文污染的子代理自评。验收不通过则打回重做或记入 `failures.md`；通过后再提交压缩结果。验收子代理属收敛位，档位见「角色与模型按档位分配」。

### 双审等待与缓存/上下文成本

前提：Anthropic 缓存按前缀精确匹配、命中即刷新、按模型×effort 隔离；TTL 因环境而异且有波动史（订阅主会话 2026-07 实测 1h、API 直连/子代理 5m），**以本机实测为准**——详见 [claude-cache-keepalive](https://github.com/ruodou233/claude-cache-keepalive)。主会话保温的唯一途径是主回合自身发请求；OpenAI 侧自动缓存零写入费，Codex 主会话无保温需求。本节心跳为 Claude 侧专用。

1. **批量送审**：一轮内多项审查攒一批送，一次等待付一次全量重读。
2. **等待期心跳（Claude 侧）**：已升级为通用 skill [`cache-keepalive`](https://github.com/ruodou233/claude-cache-keepalive)（v6，2026-07-04）——按环境分策略（订阅主会话 1h TTL：场景 B（等用户回复）为 50 分钟节拍改进循环；5m 环境用错峰拍链），节拍/上限/命中校验均以该 skill 为准，本节不再重复。作者环境另有 turn-guard hook 条件式提醒（私有自动化，不随包发布）。
3. **两笔账**：缓存过期是一次性损失；主上下文膨胀是每轮重复成本。大体量阅读必须走子代理；子代理模型选择与保温无关。
4. **噪音与缓存的优先级**：防进入（子代理准入控制）> 只追加（新指令/状态追加在尾部，不回改前缀）> 择机批量清（任务边界一次性清理/compact；新话题直接新会话）。
5. **恢复包**：长会话在 scratchpad 维护"方案摘要+当前决策+未决项+下一步"，跨会话续命只读恢复包。

### 失败要记录，不能无限重试

失败应写入 `failures.md`，至少记录：

- 任务 ID；
- 失败类型；
- 使用的方法；
- 失败表现；
- 可能原因；
- 是否值得重试；
- 下次不要再试的条件。

同一类失败连续出现时，应触发跳过、降级或请求用户确认，而不是无限重试。

### 接近上下文阈值就交接

当主代理上下文接近上限，或长任务进入下一阶段时，应生成 handoff 文档。

handoff 至少包含：

- 已完成工作；
- 进行中任务；
- 当前方案；
- 有效方法；
- 失败方法；
- 关键决策；
- 机器可读状态文件位置；
- 下一步建议；
- 未解决问题。

## 最小工作流程

### 1. 任务澄清

在开始长任务前，先判断或确认：

- 目标和完成定义；
- 输入材料和权限边界；
- 交付物格式；
- 时间、token、成本限制；
- 优先级取舍；
- 哪些动作已明确授权，哪些仍需澄清。

### 2. 清点和估算

只读取元信息：

- 文件名；
- 修改时间；
- 行数；
- 字节数；
- 可见文本粗估 token；
- 是否属于敏感或受限目录。

不要在清点阶段读取全文。

### 3. 建立 manifest 和 queue

把对象写入 `manifest.jsonl`，把任务写入 `queue.jsonl`。

每个任务应能独立重跑或跳过。

### 4. 分派执行

根据任务类型选择：

- 子代理：适合语义分析、总结、审计、比较；
- 脚本：适合确定性转换、统计、批量文件处理；
- 工具：适合读写文件、查询状态、调用外部系统；
- 主代理：只处理最终汇总、决策和用户沟通。

### 5. 回收结果

完整结果落盘，主代理只读取：

- 摘要；
- 关键证据；
- 风险；
- 建议；
- 输出路径。

### 6. 合成和确认

主代理负责去重、交叉验证、排序，并给用户明确的行动方案。

写入、合并、归档、删除、发送消息、创建外部资源等动作，用户已明确授权时按对应平台和用户偏好推进；目标不清、权限不可达或外部影响不可逆时再澄清。

## 用于升级审计的轻量模式

（配套的完整审计流程见 upgrade-audit skill：https://github.com/ruodou233/upgrade-audit ）

升级审计通常不需要完整 Python 编排器，但应吸收以下轻量治理层：

- 每次审计创建运行目录；
- 用 `manifest.jsonl` 记录被审计对象；
- 用 `queue.jsonl` 记录扫描任务；
- 大对话和大文档默认分片或分派子代理；
- 子代理完整输出保存到文件；
- 主代理只读取压缩摘要；
- 报告生成前做去重和交叉验证；
- 无人值守或纯报告模式下，写入共享记忆或修改 skill 前先给出建议；用户本轮已明确要求执行时直接推进并在完成后说明。

## 平台适配

本节为**作者实现参考（截至 2026-07）**：作者使用 Claude Code + Codex 双平台的分工细节，供你映射到自己的环境；单平台使用者可跳过，由自适应节生成自己的映射。

### 跨平台分流

任务先按两条标准在 Claude Code / Codex 之间分流，再进入各平台的工人调度：

- **耦合度**：与当前会话上下文耦合紧（边讨论边改、多轮迭代、大量隐含上下文）→ 留发起侧；自包含、可写清验收标准 → 可跨平台派发。
- **重要度**：方案制定、架构判断、终审不降档、不外包，由主代理执行。

默认方向：**所有具体执行工作**（重型调研、大块编码、批量/过夜、扫描分析等，不限于重消耗）默认派 Codex 侧（token 充裕；桌面端具备 MCP/skills/Automations/全权限本机访问，经 `~/.agents/skills/` 可读全部共享 skills 含 lark-*，协作平台操作走对应 CLI（作者环境用 lark-cli），2026-07-03 已实测）；Claude 侧只保留战略规划、指挥编排、验收/终审、会话耦合紧的重要脑力与下列排除项（2026-07-06）。

排除项——以下任务即使重消耗也不外派：中文写作/改稿与表达润色（模型偏好见 agent-memory——作者私有文档）、强交互迭代、不可逆外部副作用、安全/认证/资金相关、依赖当前登录 UI 状态的任务。

验收：跨平台派发的产出由发起侧主代理验收。被派方回报最小证据格式为「结论 + 输出文件路径 + 自检结果」，主代理只读摘要、按风险抽查，不重读全文。交付前独立验收（见核心原则）不因分流取消；agent-memory（作者私有文档）双审条目中 Claude 侧方案内审的取消只影响双审流程，不影响本节验收机制。

### Codex

Codex 可使用本地 `overnight-task` 作为执行适配器。执行前应先读取本共享规范，再读取 Codex 本地 skill 的脚本、协议和模板。

Codex 的子代理可用于上下文密集型扫描和分析。主代理保留调度、整合和最终决策。

工人模型/effort（token 充裕侧默认强档）：token 充裕 → 工人默认强档（型号/effort 以当前 `~/.codex/config.toml` 与 agent-memory.md（作者私有文档）平台映射为准），一般不为省 token 降 model；只有低价值批处理、或延迟/吞吐/可用性约束明确时才降。每次 `codex exec` 显式传参、无会话继承；按难度主要调 effort：`codex exec -c 'model_reasoning_effort="<档>"'`。预设档可建 `$CODEX_HOME/<profile>.config.toml` 后用 `-p <profile>`（当前未配置）。自动化中用绝对路径或 `CODEX_BIN`，勿假设 `codex` 在 PATH。Codex 侧无逐回合 hook 接口（AGENTS.md 为行为约定非确定性机制），确定性流程合规走共用脚本（automations/compliance/compliance-check.sh check，收尾前运行）（作者私有脚本，不随包发布；确定性合规检查思路可由你的 Agent 生成等价物）+ 每日审计兜底。

### Claude Code

Claude Code 可使用 subagent、hook、命令或本地 CLI 实现同一套治理方法。

子代理用途与档位（2026-07-06 起）：Claude 侧默认不启动执行类工人子代理——执行工作派 Codex 侧（见「跨平台分流」）。**Claude 侧例外表**（唯一权威清单，防散写漂移）：

- 允许：主代理战略/指挥/终审；收敛位（验收/聚合/评审，取强档，见角色段）；会话耦合紧的并行分析（如 wisdom-roundtable）；分流排除项任务；用户点名 `ultracode`（其多代理机制属用户显式 opt-in，不受本默认约束）。
- 遗留（维持现状 + 迁移待办）：`claude-audit` 的 Sonnet 扫描子代理；`turn_classifier.py` 的 Haiku 分类器（默认关闭）。该迁移评估记录在作者私有审计状态文件中（不随包发布）。

原「工人默认 Sonnet / Haiku 批量」规则废止。

钉精确模型版本（档位规则的明确例外）：默认按档位分配、不钉具体型号；仅当用户偏好或任务明确要求历史精确版本时例外（当前已确认例外：中文写作、改稿和表达润色优先 claude-opus-4-6）。两条路径：

- **交互会话内**：委派自定义子代理 `opus-writer`（作者环境的自定义子代理，机制可复制；源 `agents/opus-writer.md`，symlink 至 `~/.claude/agents/`；frontmatter `model` 官方支持 full model ID）。注意：子代理列表在会话启动时加载，新建/修改后需新会话才生效。
- **脚本/自动化/Codex 侧**：`~/Documents/AgentOps/bin/claude-pin -m <model_id> "<提示词>"`（作者私有脚本，不随包发布；机制=headless 调用时显式钉 full model ID + 回读 modelUsage 验证），长提示推荐 stdin 管道；默认模型 claude-opus-4-6，环境变量 `CLAUDE_PIN_MODEL` 覆盖。不加 `--bare`（bare 跳过 OAuth/keychain，订阅认证失效；官方计划未来让 `-p` 默认 bare，届时如订阅 OAuth 失效应报错提示，不得静默改用 API key 计费）。

## Skill 结构原则

（作者的 AgentOps 分发架构参考，供建立自己文档体系时借鉴。）

多 Agent 共用一套方法时，核心逻辑只维护一份，其余位置只放指针或适配层。目标不是让每个 agent 使用完全相同的工具，而是让它们共享同一套记忆、判断标准、工作方法和改进闭环；平台特有的执行细节保持为轻量、明确的适配层。这样拆是为了避免两类失败模式：**重复漂移**——同一条规则在两处各自演化出不同版本，改一处漏一处；**虚假能力**——一个 agent 看到另一平台的工具说明，误以为自己也能执行。

单源是维护规则，不是固定目录，当前有两种形态：

- **完整 skill 共用**：skill 整体放 `skills/<name>/`，两平台的 skill 目录用 symlink 指向同一份运行副本。
- **共享规范 + 薄入口**：跨入口共用的规范独立成单一文档，各平台入口只保留触发词、指针和平台工具适配。只有多个 agent 确实能执行相同命令/工具时才合并执行细节；否则保留一份共享 reference，各配一个本地 adapter。

两条护栏：薄入口不把共享文档内容抄写回来求"自包含"——那会复活多副本；不在共享源里写入指向 symlink 后自身的"本地适配器路径"，防止循环引用。

## 不要做什么

- 不要把所有平台的工具细节塞进一个巨大 active skill；
- 不要让主代理直接读取所有长对话和大文档；
- 不要把完整子代理输出反复塞回主上下文；
- 不要让平台专属工具说明造成其他 agent 的虚假能力；
- 不要把一次性偏好写成长期规则；
- 不要把未授权或目标不清的写入、删除、合并、归档、外部发送当成已授权动作；用户明确要求执行时按平台机制推进。

## 首次使用：环境自适应（由你的 Agent 执行）

本 skill 的默认参数来自作者环境，正文标注"作者环境示例"的数值仅作参考。首次使用时：

1. **只读探测**（默认可做）：探测已安装的 Agent CLI 或支持 SKILL.md 的平台、每个平台可用的只读/写入能力、是否支持子代理或并行会话、使用者声明的订阅类型或额度约束。探测一律只读，不修改任何系统配置。单平台时，工人全放本侧；多平台时，执行工人优先放 token 充裕侧，收敛位保留在主平台。
2. **写入前先征得同意**：以下动作必须先向使用者说明将做什么、写到哪，获明确同意后再执行——写配置文件、读取/扫描对话记录与 usage 数据、修改全局记忆入口、安装定时任务。
3. **生成本地配置**：把适配后的参数写入 `~/.config/agentops-skills/agent-orchestration/local-config.md`（格式参考 repo 内 `local-config.example.md`）。无法写该路径时退回 skill 目录内 `local-config.md`（已在 .gitignore）；两处都不可写则本次会话内直接使用，并提示使用者手动保存。
4. **之后每次触发**：先查上述路径，存在则以其中参数为准；环境变化（新增设备、换订阅、装新 CLI）时重跑本节。
